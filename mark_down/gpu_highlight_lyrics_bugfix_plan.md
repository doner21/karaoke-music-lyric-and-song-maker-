# GPU Rendering Engine — Highlight Lyrics Bug Fix Plan

## Problem Statement

The GPU rendering engine (`karaokeDrawerGL.js`) does not correctly render the **video preview highlight lyrics**. Although the CPU renderer (`karaokeDrawer.js`) and the React-based live preview (`LetterFillWord.jsx`) both correctly show progressive left-to-right highlight fill on lyrics, the WebGL2 path produces broken or invisible highlights in the exported video frames.

The GPU renderer is used exclusively during MP4 export via `electronExport.js` when `renderMode === 'webgl2'`.

---

## Root Cause Analysis

After thorough analysis of both renderers, **5 bugs** were identified in [`drawLyricsPageGL()`](file:///C:/Users/donald%20clark/.gemini/antigravity/scratch/karaoke-box/src/utils/karaokeDrawerGL.js#L755-L857):

### Bug 1: Glow Pass Overwrites Highlight Overlay (Critical)

The glow pass (lines 809–826) renders **all words as white text** to an FBO, blurs it, then composites the result back onto the **default framebuffer**. This composite step uses the `compositeProgram` which renders a full-screen quad — overwriting anything already on the default framebuffer at those pixels.

The highlight overlay (lines 828–856) is drawn **after** the glow pass, using `gl.SCISSOR_TEST` to clip the highlight color. However, the glow composite already wrote white-tinted pixels to the default framebuffer. The highlight overlay draws **on top** using standard alpha blending (`SRC_ALPHA, ONE_MINUS_SRC_ALPHA`), but since the glow composite already deposited opaque white text, the colored highlight blends poorly and appears washed out or invisible.

**CPU equivalent:** In `karaokeDrawer.js` (lines 219–243), the glow is applied via `ctx2d.shadowBlur`/`shadowColor` which only affects the current `fillText()` call. The highlight overlay uses `ctx2d.save()`/`clip()`/`restore()` which is a separate isolated draw — no interference.

### Bug 2: Scissor Y-Coordinate Calculation Error (Critical)

In the highlight overlay pass (lines 839–848), the scissor rectangle Y calculation is:
```javascript
const glyphTop = lineY - currentFontSize;
const glyphHeight = currentFontSize + 8; // +8 for PADDING*2
const scissorY = height - glyphTop - glyphHeight; // WebGL bottom-left origin
```

However, `buildTextVertices()` (line 517) positions glyphs as:
```javascript
const y0 = y - atlas.fontSize;  // uses atlas.fontSize, not currentFontSize
```

When font scaling occurs (lines 789–800), `currentFontSize` may differ from `atlas.fontSize` because font scaling changes the atlas used. The atlas's `fontSize` is the **actual** rendered glyph size, while `currentFontSize` is the **requested** size before atlas lookup. If these diverge (e.g., due to rounding in `getAtlas`), the scissor rect won't align with the actual glyph positions, causing the highlight to clip to the wrong vertical region.

### Bug 3: Glow Renders All Words With Same Opacity (Minor)

The glow pass (lines 813–821) applies per-word opacity (`0.8` future, `0.5` past, `1.0` current) to the base white text **inside the FBO**. The composite shader then writes this to the default framebuffer. But the glow blur doesn't differentiate — the blur mixes all words' alpha together, so past/future/current words will bleed glow into each other, producing a more uniform glow than the CPU path.

In the CPU path, `shadowBlur` is per-`fillText()` call, so each word's glow is independent.

### Bug 4: Font Atlas Uses Alpha Channel Incorrectly for Colored Text (Moderate)

The font atlas is generated by drawing white text onto a transparent canvas (line 223: `ctx.fillStyle = '#ffffff'`). The fragment shader (line 59) reads only the **alpha channel**: `float alpha = texture(u_atlas, v_texCoord).a;`

However, the Canvas 2D `fillText()` with `#ffffff` on a transparent canvas produces **premultiplied alpha** in many browsers. The alpha channel may not accurately represent the glyph shape — it may contain anti-aliasing values that are premultiplied with the white color. This can cause the highlight color overlay to appear semi-transparent or have incorrect anti-aliasing edges compared to the CPU path.

### Bug 5: Missing Blend State Reset Between Glow and Scissor Passes (Moderate)

The `drawWithGlow()` function (lines 636–693) uses the blur and composite programs, which expect the full-screen quad VAO (`quadVAO`). After compositing, the VAO is unbound (line 692), but the active texture units remain set to the FBO textures. The subsequent `drawTextGL()` calls in the highlight pass rebind the atlas texture, but if the composite program left any GL state dirty (e.g., active program, blend mode), this could cause the highlight text to render with incorrect blending.

## Proposed Changes

### Overview

The fix requires restructuring `drawLyricsPageGL()` to follow the same two-pass rendering strategy as the CPU path: first draw all base text (white, with glow), then draw highlight overlays on top with proper isolation.

---

### Component: GL Lyrics Renderer

#### [MODIFY] [karaokeDrawerGL.js](file:///C:/Users/donald%20clark/.gemini/antigravity/scratch/karaoke-box/src/utils/karaokeDrawerGL.js)

**Changes to `drawLyricsPageGL()` (lines 755–857):**

1. **Restructure rendering order** — Draw base white text with glow for ALL lines first (including per-word opacity), then draw highlight overlays for ALL lines in a second pass. This prevents the glow composite from overwriting highlights.

2. **Fix scissor Y-coordinate** — After font scaling, use the actual atlas `fontSize` (from `getAtlas()`) instead of `currentFontSize` to calculate the scissor rectangle. This ensures the scissor rect matches the glyph positions produced by `buildTextVertices()`.

3. **Use per-line glow instead of per-word glow** — Keep the current batched-per-line glow approach but render the highlight pass completely after all glow compositing is done. Specifically:
   - Pass 1 (all lines): For each line, draw all white text words with `drawWithGlow()`, exactly as currently done.
   - Pass 2 (all lines): For each line, draw highlight overlay text with scissor clipping.

4. **Reset GL state before highlight pass** — After all glow compositing is done, explicitly reset the active program, blend state, and texture bindings before starting the scissor-based highlight pass.

5. **Fix atlas premultiplied alpha** — When generating the font atlas, use `ctx.getContext('2d', { willReadFrequently: true })` and set the canvas to use non-premultiplied alpha by rendering glyphs to a separate canvas first, then reading pixel data. Alternatively, account for premultiplied alpha in the fragment shader by using `texture.r` instead of `texture.a` (since white text has R=G=B=A for non-transparent pixels).
